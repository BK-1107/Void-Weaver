# Project Void Weaver - 企业级 AI 图像生成工作站说明文档

## 1. 项目定位 (Executive Summary)
**Project Void Weaver** 是一款面向专业创作者与企业级场景的 **“沉浸式 AI 图像生成与解构工作站”**。它不仅支持 T2I (Text-to-Image) 生成，更核心的功能是将“视觉现实”解构为“可编辑代码（Prompt Tags）”。通过 LLM 的逻辑分析与多引擎绘图能力，实现对图像特征的精准控制。

---

## 2. 核心技术栈与架构 (Core Technology & Architecture)

### 2.1 后端：引擎驱动层 (Backend Engine)
- **核心框架**: **Java Spring Boot 3.2+** (JDK 17/21)。
- **网络层**: **Spring 6 RestClient** & **OkHttp3**。用于处理 AI 接口的异步长连接与流式响应。
- **并发处理**: 支持 **SSE (Server-Sent Events)** 协议，实现“深度思考”过程的实时推送。
- **安全分析**: 开启严格的 **Annotation-based Null Analysis**，代码全量适配 `@NonNull` 契约，确保生产环境零 `NPE`。

### 2.2 前端：交互工作台 (Frontend Workstation)
- **框架**: **React 18** + **TypeScript** + **Vite**。
- **状态流**: **Zustand**。管理 8 个核心模块的嵌套状态与锁定逻辑。
- **视觉层**: **Tailwind CSS** + **shadcn/ui**。实现高科技感的深色沉浸工作站。

## 3. 后端功能深度实现 (Backend Implementation Details)

### 3.1 智能现实解构 (Decipher Reality)
该功能通过 `GeminiService` 实现，将非结构化的图像数据转化为高度结构化的标签网格。
- **实现逻辑**: 
    - **Prompt Engineering**: 后端构造了一个复杂的 System Prompt，强制 Gemini 开启 **JSON Mode**。要求模型将图片拆解为 Style, Subject, Pose, Costume, Background, Composition, Atmosphere, Extra 八大维度。
    - **隐形特征提取**: 专门设计了颜色感知逻辑，后端会要求模型提取 5-7 种主导色（HEX 格式）并以 `hidden: true` 的形式存入 Atmosphere 模块，用于后续生成时的色彩一致性。
    - **类型映射**: 使用 Jackson 序列化将 AI 响应直接映射为 `AnalyzeResponse` DTO。

### 3.2 范围重构逻辑 (Scoped Refinement)
实现“保持人物不变，仅更换背景”等精准局部重写功能。
- **实现逻辑**:
    - **状态过滤**: 后端接收完整的模块列表，根据前端传回的 `locked` 状态位进行过滤。
    - **增量重写**: 仅将“未锁定”的模块与用户的自然语言指令发送给 Gemini。
    - **契约保证**: 系统要求模型必须返回全部模块（包括未修改的），以保证前端 UI 状态的连续性。

### 3.3 标签权重协议 (Tag-Weight Protocol)
为了让不同引擎理解权重，后端在 `PromptService` 中实现了一套转换协议。
- **编码格式**: 自研权重语法 `weight::text::`。
- **引擎适配器**:
    - **NovelAI 适配**: 直接输出带权重的标签字符串。
    - **Google Imagen 适配**: 后端在发送给 Google 前，会通过正则表达式识别权重。例如：`1.5::cat::` 会被后端预处理引擎自动转化为 `highly detailed cat` 等自然语言描述，以适配不支持原生权重语法的模型。

### 3.4 深度思考状态机 (Deep Thinking SSE)
这是后端最复杂的实现，位于 `ImageService.java`。它模拟了人类画师的创作过程：
1. **Phase 1: Sketch (草图)**: 调用 T2I 引擎生成初稿。
2. **Phase 2: Critique (审校)**: 将初稿再送回 Gemini，由“艺术总监”角色分析解剖学错误、构图缺陷。
3. **Phase 3: Suggestion (启发)**: 自动联想 Danbooru 等高质量风格标签。
4. **Phase 4: Style Injection (灵魂注入)**: 将“审校指令”与“风格标签”融合，构建最终的高级 Prompt。
5. **Phase 5: Final Manifestation (终稿)**: 进行 Img2Img（图生图）重绘。
- **实时推送**: 每一个 Phase 的日志都通过 `SseEmitter` 实时推送至前端，让用户感知 AI 的“思考过程”。

### 3.4 核心引擎：AI 编排与解析 (GeminiService)
- **结构化输出保证**: 在 `buildAnalyzeRequestBody` 中，后端向 Gemini 注入了极其严密的 **System Prompt**。不仅要求返回 JSON，还定义了具体的字段约束（如 ID 必须是标准的 UUID 格式，标签文本必须为英文）。通过 `responseMimeType: application/json` 选项，从协议层面确保了数据解析的稳定性。
- **动态重构逻辑**: `refineModules` 方法利用 Java Stream 流处理前端传回的 `unlocked` 模块，实现了 **“选择性上下文重写”**。后端在构造 Prompt 时会将当前的模块状态序列化为 JSON 传给 AI，要求其在保留锁定模块内容的同时，根据用户指令重构解锁模块。

### 3.5 高并发流式架构 (SSE + CompletableFuture)
- **非阻塞执行**: 在 `GenerateController` 中，流式请求（`/api/generate/stream`）采用 **`CompletableFuture.runAsync`** 异步执行。这意味着后端在处理耗时极长的 AI 生成任务时，不会占用 Servlet 容器的线程池，从而支持更高的并发。
- **长连接管理**: 
    - 显式配置 `SseEmitter` 的超时时间为 **300,000ms (5分钟)**，适配 Deep Thinking 模式下的多轮分析。
    - 通过定义多维度的事件名称（`result`, `log`, `sketch`, `error`），前端可以根据事件类型进行差异化渲染。

### 3.6 图像处理与多引擎兼容 (ImageService)
- **正则预处理器**: 后端实现了一个基于正则表达式 `(\\d+(?:\\.\\d+)?)::(.*?)::` 的预处理引擎。
    - **权重解构**: 它能自动识别标签中的权重（如 `1.5::cat::`）。
    - **自然语言转换**: 针对 Google Imagen 不支持权重语法的问题，后端会将权重转化为自然语言描述（如 `1.5` 映射为 `highly detailed`, `2.0` 映射为 `extremely detailed`），实现不同模型间的平滑适配。
- **二进制流解压**: 处理 NovelAI 请求时，由于其返回的是包含 PNG 数据的 **ZIP 压缩流**，后端利用 `ZipInputStream` 在内存中进行动态解压，并实时转换为 Base64 字符串返回前端，避免了文件落地的开销。

### 3.7 前后端契约与无状态设计 (Data & API)
- **DTO 层约束**: 整个系统由严谨的 DTO 层驱动。利用 **Jakarta Validation (`@Valid`)** 在 Controller 入口处拦截非法请求，确保传入的图片数据和 API Key 格式正确。
- **无状态架构**: 整个后端工程 **不依赖任何数据库 (H2/MySQL)**。所有的状态（API Keys, 模块数据, 历史记录）均在前端 `Zustand` 中管理。后端仅充当一个 **“智能协议转发器”**。这种架构极大地降低了运维复杂度，并原生支持容器化水平扩展。

### 3.8 健壮性与异常体系 (Error Handling)
- **自定义异常栈**: 建立了 `ApiException` 异常体系，能够根据不同的 HTTP 状态码（如 401, 429, 500）动态生成对应的错误代码（`INVALID_API_KEY`, `RATE_LIMITED` 等）。
- **Null Safety 实战**: 
    - 后端全量适配了最新的 **Java 空安全静态分析**。
    - 在处理 AI 返回的 `JsonNode` 时，通过 `path()` 方法进行防御式读取，避免了因 AI 输出字段缺失导致的空指针崩溃。
    - 在 SSE 推送环节，强制使用 `Objects.requireNonNull()` 进行校验，确保推送到前端的数据始终合法。

## 5. 后端核心面试 Q&A (Interview Preparation)

为了帮助你在面试中更好地展示项目深度，以下总结了针对本后端架构的高频考察点及回答建议：

### Q1: 为什么在项目中选择 Spring Boot 3.2+ 和 JDK 17/21，而不是更成熟的 JDK 8？
**A**: 
1. **性能与并发**: JDK 17+ 引入了更高效的垃圾回收器（如 ZGC）和更强的 JIT 编译优化。
2. **Spring 6 特性**: Spring Boot 3 基于 Spring 6，引入了原生的 `RestClient`。它提供了非阻塞、响应式的编程模型，在处理 AI 接口的长连接（如 SSE）时比传统的 `RestTemplate` 更简洁、性能更好。
3. **语言特性**: 利用了 `records` (用于 DTO)、`Switch Expression` 等新语法，极大减少了样板代码，提升了代码的可读性和类型安全性。

### Q2: 项目中为什么要特别强调 “Null Safety (空安全)”？你是如何实现的？
**A**: 
- **原因**: NullPointerException (NPE) 是 Java 开发中最常见的线上事故原因。在处理 AI 返回的不确定性 JSON 数据时，空安全尤为重要。
- **实现**:
    1. **严格静态分析**: 开启了基于 JDT 的编译器级别分析。
    2. **注解约束**: 在所有 Controller 和 Service 的参数、返回值上显式使用 `@NonNull` 和 `@Nullable`（来自 `org.springframework.lang`）。
    3. **运行时校验**: 在关键逻辑处（如 SSE 发送数据）使用 `java.util.Objects.requireNonNull()` 进行断言。
    4. **防御式编程**: 对 AI 返回的 JSON 节点使用 `path()` 而非 `get()`，防止链式调用崩溃。

### Q3: 深度思考模式的实时过程推送是如何实现的？为什么不选 WebSocket？
**A**:
- **实现**: 采用 **SSE (Server-Sent Events)** 协议。后端利用 Spring 的 `SseEmitter` 建立持久连接。当 `ImageService` 状态机执行完一个 Phase（如草图生成或审校）后，立即通过该 Emitter 推送一个事件到前端。
- **选型原因**: 
    1. **轻量化**: AI 绘图场景主要是“服务器向客户端单向推送日志”，SSE 相比 WebSocket 更轻量，基于 HTTP 协议，且自带断线重连机制。
    2. **兼容性**: 完美适配负载均衡（如 Nginx）和云平台的长连接策略，开发成本更低。

### Q4: 如何处理 AI API 调用可能导致的超长耗时或超时问题？
**A**: 
1. **连接池优化**: 使用 **OkHttp3** 自定义配置，将 `readTimeout` 针对“深度思考”模式延长至 300 秒，同时设置合理的连接池大小防止资源耗尽。
2. **异步非阻塞**: SSE 的使用本身就是一种异步反馈机制，后端在执行绘图逻辑时不会阻塞其他请求的响应。
3. **重试与降级**: 对不稳定的 AI 接口实现了异常拦截（ApiException），并提供错误日志实时推送到前端，保证用户体验不中断。

### Q5: 面对多绘图引擎（Google vs NovelAI），你是如何设计后端代码的？
**A**: 
- 采用了 **策略模式 (Strategy Pattern)** 的思想。在 `ImageService` 中，根据 `EngineType` 枚举动态调度不同的私有方法（如 `generateWithGoogleGemini` 和 `generateWithNovelAI`）。
- 同时通过 `PromptService` 作为中转层，解耦了“标签原始数据”与“引擎特定 Prompt 格式”，确保未来接入 Stable Diffusion 或 Midjourney 时，只需增加一个新的适配器方法而无需修改核心逻辑。
